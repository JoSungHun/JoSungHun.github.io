개인정보의 안전성 확보조치 기준 요약

개인정보의 안전성 확보조치 기준 - 개보법

제5조 (접근 권한의 관리)

  1. 개인정보처리시스템에 대한 접근권한은 최소한의 범위로 업무 담당자에 따라 차등 부여
  2. 인사이동 발생하면 지체없이 접근권한 변경 또는 말소
  3. 권한부여, 변경, 말소에 대한 기록 3년 보관
  4. 개인정보취급자 별로 계정 발급, 공유되지 않도록
  5. 비밀번호 작성규칙을 수립하여 적용
  6. 비밀번호 일정 횟수 이상 잘못 입력하면 접근 차단 등 기술적 조치

제6조 (접근 통제)

  1. 불법접근, 침해사고 방지를 위해 다음 기능 조치
    • 인가받지 않은 IP 접근 제한
    • 접속한 IP 분석하여 개인정보 유출 시도 탐지 및 대응
  2. 외부에서 접속하려는 경우 VPN, 전용선 등 안전한 접속수단, 인증수단 적용
  3. 개인정보처리시스템, 컴퓨터 등 단말기에 접근 통제 조치
  4. 연 1회 취약점 점검 및 보완조치
  5. 일정시간 이상 업무처리를 하지 않으면 자동 접속 차단
  6. 처리시스템이 아닌 단말기를 사용할 경우 1항 적용 x

제7조 (개인정보의 암호화)

  1. 고유식별정보, 비밀번호, 바이오정보를 송신, 전달하는 경우 암호화
  2. 비밀번호는 일방향 암호화 저장, 바이오정보는 암호화 저장
  3. DMZ에 고유식별정보를 저장하는 경우 암호화
  4. 내부망에 고유식별정보 저장 경우 암호화 미적용시 위험도 분석에 따라 결정
  5. 개인정보 암호화 할거면 안전한 암호화 알고리즘 사용
  6. 암호화된 개인정보 보관을 위해 절차 수립. 시행 해야함
  7. 업무용 단말기에 고유식별정보를 저장할 경우 암호화 소프트웨어, 안전한 알고리즘 사용후 저장

제8조 (접속기록의 보관 및 점검)

  1. 개인정보처리시스템 접속기록 1년보관, 5만명 이상의 정보처리 또는 고유식별정보 또는 민감정보 처리시 2년이상 보관, 관리
  2. 개인정보처리시스템 접속기록 월 1회 이상 점검, 다운로드시 사유 확인
  3. 접속기록 안전하게 보관하여야함

제9조 (악성프로그램 등 방지)

  1. 보안 프로그램 자동 업데이트 또는 1회 이상 업데이트 실시
  2. 보안 업데이트 공지가 있는 경우 즉시 업데이트
  3. 악성프로그램 발견시 삭제

제10조 (관리용 단말기의 안전조치)

  1. 인가받지 않은 사람이 조작 못하도록 조치
  2. 목적외 사용되지 않도록 조치
  3. 보안조치 적용

제11조 (물리적 안전조치)

  1. 개인정보를 보관하는 물리적 보관 장소를 별도로 두고 있는 경우 출입통제 절차 수립, 운영 해야함
  2. 개인정보가 포함된 서류, 보조저장매체 등은 잠금장치가 있는 안전한 장소에 보관
  3. 반출.입 통제를 위한 보안대책 마련해야 함. 별도의 개인정보처리시스템을 운영하지 않는 경우 적용 x

제12조 (재해 · 재난 대비 안전조치)

  1. 재해 · 재난 발생시 개인정보처리시스템 보호를 위한 위기대응 매뉴얼 등 대응절차 마련하고 정기적으로 점검해야함
  2. 재해 · 재난 발생시 개인정보처리시스템 백업 및 복구를 위한 계획을 마련해야함

제13조 (개인정보의 파기)

  1. 개인정보를 파기할 경우 다음 중 어느 하나의 조치를 해야함
    • 완전파괴(소각.파쇄 등)
    • 전용 소자장비를 이용하여 삭제
    • 데이터가 복원되지 않도록 초기화 또는 덮어쓰기
  2. 개인정보의 일부만 파기하는 경우, 1항의 방법이 어려운 경우 다음을 조치
    • 전자적 파일인 경우 : 삭제한 후 복구 및 재생되지 않도록 관리 감독
    • 기록매체인 경우 : 해당 부분을 마스킹, 천공 등으로 삭제

개인정보의 안정성 확보조치 기준을 조금 더 쉽게 외우기 위해 요약한 내용입니다.

자세하고 정확한 내용은 개인정보의 안전성 확보조치를 참고해야합니다.

Webhacking.kr 43번 문제

문제를 보면 웹쉘을 업로드하고 /flag 파일을 읽으라고 쓰여있다.

우선 웹쉘을 준비한다.

이제 이 파일을 업로드해야 하는데 web.php 라는 이름으로 저장한 후 파일을 업로드하면 wrong type이라고 메시지가 나온다. 메시지를 보면 type을 검사한다는 것을 알 수 있다.

그래서 jpg 확장자로 업로드 해봤더니 Done!이라는 메시지가 뜨고 친절하게 어디에 업로드 되었는지 까지 나온다.

업로드한 파일의 확장자를 확인하여 차단하는 경우일 수도 있고, http accept 헤더를 통해 차단할 수도 있다. 업로드 한 후 버프수트로 잡아서 확인해본다.

버프수트로 확인해 보면 accept 헤더로 확인하고 있다.

버프수트를 사용하여 업로드하는 파일의 Content-Type을 Accept에 맞게 변경한 후 보내면 업로드 되는것을 확인할 수 있다.

업로드된 링크를 들어가 보면 flag값이 나온다.

webhacking.kr 41번 문제

문제 소스코드를 보면 다음과 같다.

업로드 파일의 이름을 받아서 ./uploaddir/파일이름 으로 파일을 작성하고 flag를 입력한다.

하지만 uploaddir의 이름을 알수없으므로 알아내서 읽으면 문제가 풀릴것 같다.

., <, >, / 를 공백으로 치환하고 있다.

코드 위쪽을 보면 에러를 표시하게 하고 있다.
하지만 밑에 fopen, fwirte, fclose 함수 앞에 @를 사용하여 오류메시지를 표시하지 않도록 했다.

하지만 위에 copy함수앞에 @를 쓰지않아 copy함수에 대한 오류는 출력이 된다.

파일을 선택하지 않고 업로드해 보겠다.

copy함수에 대한 에러메시지를 출력하는것을 확인할 수 있다.

copy함수에 업로드 디렉터리에 대한 이름이 나오도록 아주 긴 값을 입력해 본다.

디렉토리의 이름이 출력된다. 이제 정상적인 파일을 업로드하고 파일을 읽어서 flag값을 얻을 수 있다.

Webhacking.kr 6번 문제

6번 문제 소스코드는 다음과 같다.

user 쿠키가 없다면 $val_id=”guest”, $val_pw=”123qwe”로 값을 주고 base64로 20번 인코딩한 후, 1은 !, 2는 @, 3은 $, 4는 ^, 5는 &, 6은 *, 7은 (, 8은 )로 변환하고 각각 user, password 쿠키에 값을 넣는다.

그 후 각각에 맞게 다시 숫자로 변환해 준 후 base64로 20번 디코딩한다. 디코딩과정을 거친후 id가 “admin”, pw가 “nimda”이면 solve함수가 호출된다.
admin, nimda값을 base64로 20번 인코딩한 후, 숫자를 위와 같이 바꿔서 쿠키에 입력하면 문제가 풀릴것으로 보인다.

base64를 20번이나 돌리고 숫자를 하나하나 바꿔주기 힘드므로 파이썬으로 간단하게 코딩을 해서 문제를 해결해 본다.

파이썬을 돌린결과를 쿠키에 입력하면 문제가 풀린다.

Webhacking.kr 1번문제

아래쪽 php 코드를 보면 user_lv 쿠키 값이 5보다 크고 6이상이면 1로 값을 바꾸고, 5보다 크면 solve 함수가 호출된다. 따라서 5보다크고 6보다 작은 숫자로 user_lv 쿠키값을 변경해 주면 문제가 풀린다.

크롬 확장프로그램인 EditThisCookie를 사용하여 쿠키값을 변경해주면 문제가 풀린다.
